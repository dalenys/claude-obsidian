## The throughline: audit-driven release discipline

The defining process change of this arc is that **every minor release is gated by a pre-push audit**, and the audit's findings drive a same-day patch release:

- v1.7 audit surfaced **BLOCKER B1** (data-egress consent gap) — code had gone _worker → commit_ with no separate verifier pass. The fix was both the `--allow-egress` flag _and_ a new `agents/verifier.md` read-only pre-commit specialist, so the class of bug can't slip through again.
- v1.8.0 pre-push audit (YELLOW: 0 BLOCKER / 4 HIGH) → **v1.8.2** closes all four → GREEN.
- v1.9.0 pre-public-promotion audit (91.6) → **v1.9.1** closes 6/6 HIGH+MEDIUM → ~94, GREEN.

The audits used the [[10-Principle Thinking Framework]] as their phase structure — which is _why_ the framework then shipped as code in v1.9.0. The release process generated its own product feature.

## Known gaps found while filing

- **`flock` portability bug — found _and fixed_ in this commit.** Both `scripts/allocate-address.sh` and the meta-lock in `scripts/wiki-lock.sh` hard-depended on `flock`, which macOS does not ship, so address allocation _and_ multi-writer locking silently failed on a stock Mac. Fixed by adding an atomic-`mkdir` spinlock fallback to both (the `flock` path is unchanged on Linux); the full `make test` suite now passes on macOS for the first time. This session's pages were hand-allocated (c-000003…c-000007) before the fix landed; `allocate-address.sh --rebuild` independently confirms next=8.
- The wiki index/log/overview were ~2 months stale relative to the code before this note.
